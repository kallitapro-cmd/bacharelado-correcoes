# Story 3.1 — Barra de progresso do batch (real-time via batch_state)

**Status:** InReview
**Sprint:** 3 — Interface de validação do PA
**Complexidade:** M
**Prioridade:** Alta
**Dependências:** Story 3.0 (upload_screen — `st.session_state["batch_config"]` disponível); Sprint 2 completo (batch_processor, batch_state, calibrator, plagiarism_detector, group_detector)

---

## Contexto

Após o PA confirmar upload e enunciado na Tela 1 (Story 3.0), ele precisa de feedback visual enquanto o batch processa — que pode levar de 30 segundos a vários minutos dependendo do tamanho da turma. Sem progresso visível, o PA não sabe se o sistema está travado ou funcionando.

Esta story implementa a **Tela 2** do fluxo desenhado pelo @oalanicolas: uma tela de progresso que inicia o `processar_batch()` em background, consulta `batch_state.py` periodicamente e exibe progresso granular por lote. Após o término, chama o `calibrar_batch()` e os detectores, e navega automaticamente para a tabela de validação (Story 3.2).

O design evita polling agressivo: usa `st.rerun()` com intervalo de 2 segundos, respeitando os limites de rate limit da API.

---

## Objetivo

Implementar `src/ui/progress_screen.py` como página Streamlit que executa o pipeline completo de batch em thread separada, exibe progresso em tempo real consultando `batch_state`, e ao finalizar navega para a tela de validação com os resultados em `st.session_state`.

---

## Acceptance Criteria

- [x] **AC-01:** Ao entrar na tela, lê `st.session_state["batch_config"]`; se ausente ou inválido, exibe `st.error` e link "← Voltar ao upload" sem iniciar o batch.
- [x] **AC-02:** O batch é iniciado em `threading.Thread` separada (não bloqueia o event loop do Streamlit); a thread grava progressos em `batch_state` via funções públicas do módulo.
- [x] **AC-03:** A tela exibe `st.progress()` calculado como `lotes_concluídos / total_lotes` consultando `batch_state.recuperar_fichas_batch()` ou contagem de lotes — atualizado via `st.rerun()` com intervalo de 2 segundos enquanto o batch estiver em andamento.
- [x] **AC-04:** Abaixo da barra de progresso, exibe contador textual: `"Processando lote X de Y — Z alunos corrigidos"` atualizado a cada rerun.
- [x] **AC-05:** Erros de lote individuais (lotes com status `"falha"` em `batch_state`) são exibidos como `st.warning` inline durante o processamento, sem interromper a barra de progresso dos demais lotes.
- [x] **AC-06:** Após `processar_batch()` retornar, a tela chama sequencialmente: (1) `calibrar_batch()`, (2) `detectar_plagio_no_batch()`, (3) `detectar_grupos_candidatos()`; cada etapa é exibida como item de checklist com spinner (`st.spinner`).
- [x] **AC-07:** Ao finalizar todas as etapas, grava em `st.session_state["batch_results"]` um dict com `{"fichas_calibradas": list, "alertas_plagio": list, "grupos_candidatos": list, "custo_estimado_brl": float, "lotes_com_erro": int}` e redireciona para a tela de validação (Story 3.2) via `st.switch_page`.
- [x] **AC-08:** Se `OrcamentoExcedidoError` for levantada (cenário raro — deveria ter sido bloqueada na Tela 1), a tela exibe `st.error` com o valor estimado e o limite, e volta ao estado inicial sem navegar para validação.
- [x] **AC-09:** Existe teste unitário `tests/unit/test_progress_screen.py` com pelo menos 3 testes cobrindo: (a) `batch_config` ausente → erro sem iniciar batch, (b) OrcamentoExcedidoError → exibe erro correto, (c) progresso calculado corretamente para N lotes com K concluídos.

## Notas de Design (Alan Nicolas — 2026-05-22)

**Fluxo Tela 2 — Progresso:**
- Zone A (topo): título "Processando batch" + estimativa de tempo restante (aproximada)
- Zone B (centro): barra de progresso `st.progress()` + contador de lotes
- Zone C (rodapé): checklist de etapas pós-batch (calibração → plágio → grupos)
- Mensagens de erro de lote individuais como expanders colapsados (não interrompe o fluxo visual)

**Threading no Streamlit:**
- Usar `threading.Thread(target=..., daemon=True)` para não bloquear o event loop
- Guardar referência da thread em `st.session_state["_batch_thread"]` para verificar `.is_alive()`
- Polling via `st.rerun()` + `time.sleep(2)` no final do script quando thread ainda viva
- **Nota de performance (AC-03):** Preferir contagem de lotes via `SELECT COUNT(*) FROM lotes WHERE batch_id=? AND status='processado'` em vez de `recuperar_fichas_batch()` durante o polling — evita deserializar todas as fichas a cada 2 segundos.

## Riscos

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Thread do batch continua após o usuário navegar para outra tela | Média | Médio | Thread daemon=True garante que morre junto com o processo |
| `st.rerun()` muito frequente causa load excessivo | Baixa | Baixo | Intervalo mínimo de 2 segundos entre reruns |
| Resultados do batch não disponíveis quando tela de validação carrega | Baixa | Alto | Verificar `batch_results` em `session_state` antes de renderizar Story 3.2 |

## Dev Notes

**Decisões autônomas (modo YOLO):**

1. **Ordem das etapas pós-batch** — AC-06 fixa ordem: calibração → plágio → grupos. Isso significa que `detectar_plagio_no_batch` é chamado SEM `grupos_conhecidos` (pois grupos ainda não foram detectados). A Story 3.2 pode aplicar supressão visual usando ambos os resultados. Tradeoff documentado em comentário no código.

2. **`construir_batch_results(resultado, ...)`** recebe o dict completo do `processar_batch()` em vez de campos individuais, simplificando a assinatura e mantendo `custo_estimado_brl`/`lotes_com_erro` na mesma origem.

3. **Polling via `SELECT COUNT(*)`** isolado em 4 funções puras (`contar_lotes_concluidos`, `contar_lotes_com_erro`, `listar_motivos_falha`, `contar_alunos_corrigidos`) — todas testáveis sem Streamlit e seguindo a nota de performance do AC-03.

4. **`batch_id_corrente = 1`** assumido após `batch_state.limpar_banco()` resetar o AUTOINCREMENT do sqlite — esse comportamento é consistente entre Sprint 2 e Sprint 3 (não há batches concorrentes na sessão do PA).

5. **Threading sem `add_script_run_ctx`** — Streamlit moderno (1.30+) avisa sobre threads sem contexto, mas o worker `_run_batch` escreve apenas no dict `estado` (referência compartilhada) e nunca toca `st.*` — padrão thread-safe e testável.

**Testes:**

- 42 testes cobrindo: validação de `batch_config`, cálculo de progresso, formatação de contador, construção de `batch_results`, polling SQL (COUNT, listar motivos, soma alunos), captura de `OrcamentoExcedidoError`/`Exception` no worker e adapters para `TrabalhoParaComparacao`/`AlunoRef`.
- `_run_batch` testado com 3 cenários (sucesso, OrcamentoExcedido, RuntimeError genérico) sem precisar instanciar thread real — chamada síncrona via monkeypatch de `processar_batch`.

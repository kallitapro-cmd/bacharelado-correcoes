## Contexto

O pipeline de correção em batch (Story 2.2) gera uma lista de `FichaCorrecao` com notas individuais por aluno. Antes de o Professor Assistente receber o resultado final, as notas precisam passar por uma etapa de calibração estatística: o sistema analisa o ranking do batch inteiro e aplica duas regras inegociáveis definidas nas heurísticas do squad `bacharelado-correcoes`.

**Regra top 10% (H-CAL-01):** os melhores `ceil(0.10 × N_alunos)` alunos do batch podem receber nota máxima (até 10.0). Isso evita que o sistema seja sistematicamente conservador nos melhores trabalhos.

**Regra cap 9 (H-CAL-02):** todos os demais alunos ficam com nota limitada a 9.0, reservando a faixa 9.0–10.0 exclusivamente para quem se destacou no ranking comparativo.

A calibração é feita com uma **única chamada ao Claude Sonnet** (`claude-sonnet-4-6`) por batch, conforme ADR-003. O Sonnet é usado (em vez do Haiku) porque a tarefa é de raciocínio comparativo entre alunos — não de correção individual — e ocorre apenas 1 vez por batch, tornando o custo marginal justificado (ver ADR-003, tabela de parâmetros). O custo dessa chamada já está contabilizado na fórmula do ADR-005 e é estimado antes do início do batch pelo `batch_processor.py` (Story 2.2).

A pre-classificação do top 10% é realizada em Python puro antes da chamada à API, de modo que o Sonnet recebe apenas o ranking já estruturado e devolve os ajustes finais de forma determinística (`temperature=0`).

## Regra top 10% e cap 9 — comportamento esperado

Para batch de 20 alunos:

- `top_n = ceil(0.10 × 20) = 2 alunos`
- Os 2 alunos com maior `nota_a1` podem ter `nota_a1` entre 9.0 e 10.0
- Os outros 18 ficam com cap: se `nota_a1 > 9.0`, é reduzida para 9.0
- Caso notas empatadas no limiar do top 10%: incluir todos os empatados no top

**Exemplo numérico (N=10):**

| Aluno | Nota bruta | top_n=1? | Nota após cap |
|-------|-----------|----------|---------------|
| RA-001 | 9.8 | Sim (maior) | 9.8 (mantida) |
| RA-002 | 9.3 | Não | 9.0 (reduzida) |
| RA-003 | 8.7 | Não | 8.7 (inalterada) |
| RA-004 | 7.5 | Não | 7.5 (inalterada) |

**Caso empate no limiar (N=10, 2 alunos com mesma nota na 1a posição):**

- `top_n = ceil(0.10 × 10) = 1`
- Notas: RA-001=9.8, RA-002=9.8 (empate no topo)
- Ambos incluídos no top porque estão empatados no limiar → ambos podem ter nota > 9.0

## Notas de Implementação

**Localização do módulo:** `src/batch/calibrator.py`

**Assinatura da função principal:**
```python
def calibrar_batch(fichas: list[FichaCorrecao], metadata: dict) -> list[FichaCorrecao]:
    ...
```

**Fluxo interno esperado:**
1. Calcular `top_n = math.ceil(0.10 * len(fichas))`
2. Ordenar fichas por `nota_a1` decrescente
3. Identificar nota limiar do top_n (incluir empatados)
4. Montar payload ao Sonnet: lista de `{ra, nota_a1}` em ordem de ranking (sem nomes)
5. Chamar Sonnet com `temperature=0`, `max_tokens=4096`, `cache_control` no system prompt, `timeout=API_TIMEOUT`
6. Receber ranking ajustado e mesclar com fichas originais
7. Aplicar cap 9: qualquer aluno fora do top com nota > 9.0 é reduzido a 9.0
8. Retornar lista de fichas com notas calibradas

**Constantes:**
```python
MODELO_CALIBRACAO = "claude-sonnet-4-6"
TEMPERATURE_CALIBRACAO = 0
MAX_TOKENS_CALIBRACAO = 4096
API_TIMEOUT = 90
```

**Padrão de log (sem PII):**
```python
log_action("inicio_calibracao", f"batch de {len(fichas)} alunos, 1 chamada Sonnet")
```

**Referência de padrão `cache_control`:** reutilizar exatamente o mesmo padrão de `cache_control: {"type": "ephemeral"}` no system prompt já definido em `clone_client.py` (V9 do ADR-002).

**Tratamento de empate:** usar `nota_limiar = sorted(notas, reverse=True)[top_n - 1]` e incluir todos os alunos com `nota_a1 >= nota_limiar` no conjunto do top 10%.

```

  - 3 alunos fora do top 2 com notas 9.1, 9.2, 9.5

  - Todos os 3 alunos fora do top têm nota final = 9.0
```

```
  - Tanto RA-001 quanto RA-002 mantêm nota > 9.0 (ambos no top por empate)
```

## Riscos

- **Risco 1:** `FichaCorrecao` pode ainda não ter schema definitivo consolidado da Story 2.2. Mitigação: verificar o schema existente em `src/batch/` antes de implementar; se necessário, coordenar com a Story 2.2 para alinhar o campo `nota_a1`.
- **Risco 2:** Resposta do Sonnet pode retornar ranking em formato inesperado. Mitigação: validar JSON de resposta com Pydantic antes de mesclar com fichas originais; lançar exceção tipada em caso de formato inválido.
- **Risco 3:** Batch vazio (`len(fichas) == 0`). Mitigação: checar `if not fichas` no início da função e retornar lista vazia sem chamar a API.

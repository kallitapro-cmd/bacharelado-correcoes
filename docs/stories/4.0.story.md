# Story 4.0 — Manual do PA

**Status:** Ready
**Sprint:** 4 — Documentação e Deploy
**Complexidade:** M
**Prioridade:** Alta — pré-requisito para entrega institucional; o sistema não tem valor operacional sem o PA saber usá-lo
**Dependências:** Sprint 3 completo (Stories 3.0–3.3 Done); `docs/guides/deploy.md` existente; ADR-004, ADR-005, ADR-006

---

## Contexto

O Corretor Acadêmico está funcionalmente completo após o Sprint 3 — upload, processamento, validação e exportação implementados e testados. Porém, o sistema só tem valor institucional real se o PA (Professor Assistente) souber usá-lo de forma autônoma, sem depender do desenvolvedor para cada operação de rotina.

O PA é o único usuário operacional do sistema. Seu perfil: docente de nível técnico-pedagógico, fluente em planilhas Excel, familiarizado com Moodle/AVA, mas sem necessidade de conhecer Python, APIs ou infraestrutura cloud. O Manual deve falar a língua do PA — fluxo de trabalho concreto, capturas de tela descritivas, sem jargão técnico.

A ausência de documentação adequada cria três riscos concretos:

1. **Erro de upload** — planilha com colunas erradas → batch não inicia → PA perde tempo tentando descobrir o formato
2. **Mal-uso dos alertas de plágio** — PA não entende que os alertas são informativos (ADR-006) e aplica penalização automática sem revisar → decisão indevida
3. **Perda de dados por encerramento de sessão** — PA fecha a aba antes de exportar → resultados perdidos (ADR-004: sem persistência cross-session)

O Manual deve, portanto, ir além de um "guia de uso" genérico: precisa prevenir ativamente esses três cenários de erro e ancorar o PA nas garantias inegociáveis do sistema.

---

## Objetivo

Produzir `docs/guides/manual-pa.md` — documento Markdown completo que cobre o fluxo operacional de ponta a ponta, os erros mais comuns com solução passo a passo, as garantias do sistema (privacidade, custo, decisão do PA), e um FAQ para dúvidas recorrentes esperadas em contexto acadêmico real.

O documento deve ser auto-suficiente: um PA sem treinamento prévio deve conseguir completar seu primeiro batch com sucesso lendo apenas este Manual.

## Acceptance Criteria

### Estrutura e Cobertura de Fluxo

- [ ] **AC-01:** O documento contém uma seção "Visão Geral" com diagrama ASCII do fluxo de 4 telas:
  ```
  Tela 1 (Upload) → Tela 2 (Progresso) → Tela 3 (Validação) → Tela 4 (Exportação)
  ```
  Cada tela tem uma linha descritiva do que o PA faz nela e do que o sistema faz em background.

- [ ] **AC-02:** A seção "Passo a Passo" cobre as 4 telas na sequência exata do fluxo, com no mínimo 3 sub-passos por tela. Para cada tela: o que o PA deve fazer, o que o sistema exibe em resposta, e qual é a condição de saída (o que habilita o avanço para a próxima tela). Telas cobertas:
  - **Tela 1 — Upload:** formato da planilha (colunas obrigatórias: `ra`, `resposta`; opcionais: `nome`, `turma`), inserção do enunciado, preenchimento de metadados opcionais, leitura da estimativa de custo, clique em "Iniciar correção"
  - **Tela 2 — Progresso:** leitura da barra de progresso por lote, o que os status `processando` / `concluído` / `erro` significam, quando o botão "Ver resultados" fica habilitado
  - **Tela 3 — Validação:** como interpretar as colunas da tabela (RA, Notas, Δ, Status, Alertas), como usar os filtros rápidos, como aprovar individualmente vs. aprovação em lote, como revisar um aluno com alerta (Progressive Disclosure L1 e L2), as três ações disponíveis (✓ Manter nota / ✏️ Editar nota / ✗ Revisão manual), quando o botão "Exportar" fica habilitado
  - **Tela 4 — Exportação:** leitura do resumo de conferência, preview das 10 primeiras linhas, download do arquivo Excel, opção "← Corrigir mais alunos"

- [ ] **AC-03:** O documento explica o **formato obrigatório da planilha** com um exemplo de tabela Markdown mostrando uma planilha válida (cabeçalhos + 3 linhas de exemplo com dados fictícios). Inclui lista explícita de colunas obrigatórias e opcionais.

- [ ] **AC-04:** O documento explica o **Progressive Disclosure de alertas** em linguagem não-técnica:
  - O que significa 🔴 (plágio severo ≥ 85%), 🟡 (plágio moderado 70–84%), 👥 (grupo detectado)
  - Como abrir o painel de detalhes (clicar "👁 Revisar")
  - Como ver os pares de plágio (clicar "Ver detalhes de plágio")
  - Como ver os membros do grupo (clicar "Ver detalhes de grupo")
  - A garantia explícita: **o sistema não altera notas automaticamente** — toda decisão é do PA

- [ ] **AC-05:** O documento inclui a seção **"Erros Comuns e Soluções"** com no mínimo 8 cenários de erro, cada um com: (a) sintoma observável no sistema, (b) causa provável, (c) solução passo a passo. Cenários obrigatórios:
  1. Planilha sem coluna `ra` → erro de validação na Tela 1
  2. Planilha com coluna `ra` com formato incorreto (ex: RA com letras, espaços) → erro
  3. Enunciado vazio → botão "Iniciar correção" desabilitado
  4. Estimativa de custo acima do limite (MAX_COST_BRL) → botão bloqueado
  5. Lote com erro na Tela 2 (número de "Lotes com erro" > 0) → o que fazer
  6. PA fecha a aba antes de exportar → dados perdidos (sem solução técnica — é um comportamento esperado por design; documentar claramente)
  7. Aluno com alerta sem decisão → botão "Exportar" bloqueado na Tela 3
  8. Arquivo Excel não abre corretamente → formatos compatíveis (Excel, LibreOffice, Google Sheets)

- [ ] **AC-06:** O documento inclui a seção **"Alertas de Plágio — Guia de Decisão"** com orientação pedagógica para as três ações do PA:
  - **✓ Manter nota:** quando usar (ex: aluno revisou o trabalho depois, similaridade justificada pelo tema)
  - **✏️ Editar nota:** quando usar (ex: PA decide penalizar parcialmente baseado nas evidências)
  - **✗ Revisão manual:** quando usar (ex: caso grave que precisa de processo disciplinar formal, ou quando PA precisa de mais tempo para decidir)
  - Lembrete explícito: "Revisão manual" não cancela a exportação — o aluno é exportado com a flag `Revisão Manual = Verdadeiro` para que a coordenação saiba que aquele registro precisa de acompanhamento

- [ ] **AC-07:** O documento inclui a seção **"Privacidade e Segurança"** explicando em linguagem simples as garantias do sistema (baseada em ADR-004):
  - Os dados dos alunos existem apenas durante a sessão aberta — ao fechar o navegador, tudo é apagado
  - Os nomes dos alunos **nunca são enviados à API de correção** (apenas RA e texto do trabalho)
  - O arquivo Excel é gerado localmente no servidor e disponibilizado para download imediato — não é armazenado
  - Orientação explícita: fazer o download do Excel **antes de fechar a aba**
  - Orientação sobre o log de auditoria: o botão "Baixar log da sessão" na sidebar registra as ações do PA durante a sessão (para fins de auditoria institucional)

- [ ] **AC-08:** O documento inclui a seção **"Custo e Orçamento"** explicando em linguagem simples (baseada em ADR-005):
  - O sistema usa a API Anthropic, que cobra por tokens processados
  - A estimativa de custo é exibida na Tela 1 **antes** do batch iniciar — o PA não é surpreendido com custos inesperados
  - O limite padrão (R$ 15,00 por batch) cobre folgadamente uma turma de até 120 alunos
  - Como ajustar o limite para turmas grandes (contato com o administrador do sistema para atualizar `MAX_COST_BRL` em secrets)
  - O custo real costuma ser significativamente menor que a estimativa (estimativa é conservadora)

- [ ] **AC-09:** O documento inclui a seção **"FAQ"** com no mínimo 10 perguntas e respostas cobrindo dúvidas reais esperadas em contexto acadêmico. Perguntas obrigatórias:
  1. "Posso corrigir duas turmas no mesmo dia?" (resposta: sim, mas são sessões separadas)
  2. "O que acontece se a internet cair durante o processamento?"
  3. "O sistema detecta uso de IA generativa (ChatGPT, etc.)?"
  4. "Posso desfazer uma aprovação em lote?"
  5. "O que é o campo 'Observações PA'?"
  6. "A planilha exportada é compatível com o sistema da coordenação?"
  7. "Posso processar menos de uma turma inteira (ex: só os alunos que faltaram)?"
  8. "O que fazer se um aluno entregou dois arquivos?"
  9. "Quanto tempo leva o processamento?"
  10. "Os alunos sabem que foram monitorados por detecção de plágio?"

- [ ] **AC-10:** O documento tem no topo um **"Índice de Navegação Rápida"** com links âncora Markdown para cada seção principal — permitindo que o PA salte diretamente para a seção de interesse sem rolar o documento inteiro.

- [ ] **AC-11:** O tom do documento é **consistentemente não-técnico**: sem menção a Python, Streamlit, API, JSON, `session_state`, `st.secrets`, ou qualquer termo de desenvolvimento de software. Termos técnicos que precisam aparecer (ex: "batch") são definidos na primeira ocorrência entre parênteses: "Lote de correção (batch)".

- [ ] **AC-12:** O documento não contém informações falsas ou contraditórias em relação à implementação real dos Sprints 1–3. Qualquer afirmação sobre comportamento do sistema deve ser verificável nos arquivos de código ou stories existentes.

## Notas de Design

**Público-alvo:** Professor Assistente (PA) — docente universitário com fluência básica em informática, sem background técnico em desenvolvimento de software.

**Princípio de redação:** Cada seção responde a uma pergunta real que o PA teria. Nenhum parágrafo existe apenas para completar a documentação — todo conteúdo deve ter utilidade direta e imediata para o PA operando o sistema.

**Tom:** Direto, respeitoso, sem infantilizar. Equivalente ao tom de um manual de aparelho eletrônico de qualidade — assume que o leitor é inteligente mas não conhece os internos do sistema.

**Estrutura das seções de erro:** Problema → Causa → Solução. Sem culpar o usuário. Sempre com ação concreta ao final.

**Referências cruzadas internas:**
- ADR-004 → Seção "Privacidade e Segurança" (retenção zero, nomes não enviados à API)
- ADR-005 → Seção "Custo e Orçamento" (estimativa conservadora, hard limit)
- ADR-006 → Seção "Alertas de Plágio" e "Guia de Decisão" (caráter informativo, não-punitivo automático)
- Stories 3.0–3.3 → "Passo a Passo" (comportamento exato de cada tela)

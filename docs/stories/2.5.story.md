# Story 2.5 — Implementar `group_detector.py` — Detecção de grupos por sobreposição de nomes

**Status:** Done
**Sprint:** 2 — Pipeline de batch de correção
**Complexidade:** S
**Prioridade:** Baixa
**Dependências:** 2.1 (Aluno schema), 2.4 (plagiarism_detector — integração com flags)

---

## Contexto

O ADR-006 (`docs/decisions/ADR-006-plagio-policy.md`) formaliza a política de detecção de plágio do sistema. Na seção "Comportamento para Grupos", o ADR registra:

> "Falso positivo aceito conscientemente — o custo de ignorar manualmente alguns pares é menor que o custo de implementar suporte a grupos antes de validar a UX."

E no roadmap futuro:

> "Adicionar coluna opcional `grupo` na planilha → sistema agrupa alunos e suprime flags dentro do mesmo grupo declarado."

Esta story implementa um mecanismo **complementar e intermediário**: ao invés de depender de uma coluna declarada na planilha, o `group_detector` infere candidatos a grupo por evidência textual — buscando padrões de auto-referência dentro dos próprios trabalhos ("eu e Bruno fizemos...", "nossa equipe com Ana...") e verificando sobreposição de nomes mencionados entre alunos.

O resultado é passado **opcionalmente** ao `plagiarism_detector` para suprimir flags de pares que são candidatos ao mesmo grupo, reduzindo falsos positivos sem modificar notas ou decisões automáticas.

A Story 2.4 (plagiarism_detector) é pré-requisito pois esta story modifica o assinatura de `detectar_plagio_no_batch()` para aceitar o parâmetro `grupos_conhecidos`.

---

## Objetivo

Implementar `src/batch/group_detector.py` — heurística para identificar candidatos a trabalhos em grupo com base em sobreposição de nomes de autores mencionados dentro dos próprios trabalhos. O detector complementa o `plagiarism_detector` sem substituí-lo, e passa seus resultados como contexto opcional para suprimir flags de pares dentro do mesmo grupo candidato.

## Acceptance Criteria

- [x] **AC-01:** `extrair_candidatos_grupo(texto: str, alunos: list[Aluno]) -> list[str]` retorna lista de RAs de alunos que são mencionados (por nome) dentro do `texto` — usando `difflib.get_close_matches` com cutoff 0.75 para tolerar variações de grafia (ex.: "Rodrigo" encontra "Rodrigo Ferreira" cujo nome está no cadastro). **Regra de ambiguidade:** quando `get_close_matches` retorna mais de 1 candidato para o mesmo token de nome, o token é descartado da lista de membros e a ocorrência é registrada nas `evidencias` como `"Nome ambíguo: N alunos com nome similar encontrados"` — o grupo resultante recebe confiança `"baixa"` automaticamente.
- [x] **AC-02:** `detectar_grupos_candidatos(trabalhos: list[Trabalho], alunos: list[Aluno]) -> list[GrupoCandidato]` retorna lista de `GrupoCandidato` onde cada objeto contém: `membros: list[str]` (RAs dos integrantes), `evidencias: list[str]` (trechos que justificam o agrupamento), `confianca: Literal["alta", "media", "baixa"]`. **Regra de tamanho:** grupos candidatos com mais de 4 membros são automaticamente rebaixados para confiança `"baixa"`, independente da reciprocidade de menções — grupos reais de 5+ integrantes em trabalho acadêmico são raros e exigem revisão manual do PA.
- [x] **AC-03:** Nível de confiança segue a regra:
  - `"alta"` — 2 ou mais alunos do grupo candidato se mencionam mutuamente (A menciona B e B menciona A)
  - `"media"` — apenas um aluno menciona o(s) outro(s), sem reciprocidade
  - `"baixa"` — apenas um padrão de grupo genérico foi detectado sem correspondência nominal direta com alunos do cadastro
- [x] **AC-04:** O detector NÃO modifica `nota` nem `flags` das fichas dos alunos — retorna apenas `list[GrupoCandidato]` sem efeitos colaterais nas estruturas de entrada
- [x] **AC-05:** `detectar_plagio_no_batch()` em `src/batch/plagiarism_detector.py` passa a aceitar parâmetro opcional `grupos_conhecidos: list[GrupoCandidato] | None = None`; quando informado, pares que pertencem ao mesmo `GrupoCandidato` têm suas flags suprimidas (não aparecem em `list[ParPlagio]`); quando `None`, comportamento idêntico ao atual
- [x] **AC-06:** Todo processamento é local, sem I/O externo, sem chamada à API Anthropic (ADR-004 — processamento 100% local)
- [x] **AC-07:** `pytest tests/unit/test_group_detector.py` passa com mínimo 5 testes sem mocks

## Notas Técnicas

### Estrutura de dados

```python
from typing import Literal
from dataclasses import dataclass, field

@dataclass
class GrupoCandidato:
    membros: list[str]          # lista de RAs (str)
    evidencias: list[str]       # trechos de texto que justificam
    confianca: Literal["alta", "media", "baixa"]
    pares_suprimidos: list[tuple[str, str, float]] = field(default_factory=list)
    # (ra_a, ra_b, similaridade) — pares que seriam flag de plágio mas foram suprimidos por este grupo
    # permite ao PA ver o que o sistema decidiu ocultar (visibilidade para tabela Sprint 3)
    razao_confianca: str = ""
    # texto legível para o PA: ex. "Ana mencionou Carlos e Carlos mencionou Ana"
    status_revisao: Literal["pendente", "confirmado_pa", "rejeitado_pa"] = "pendente"
    # campo previsto para Sprint 3: PA confirma, rejeita ou deixa pendente o agrupamento
```

### Padrões de detecção textual (sugestão de implementação)

O `group_detector` usa dois mecanismos complementares:

**Mecanismo 1 — Pattern de auto-referência explícita:**
Expressões regulares que capturam menções nominais na primeira pessoa:
- `"eu e [Nome]"`, `"[Nome] e eu"`
- `"com [Nome]"`, `"junto com [Nome]"`
- `"nossa equipe"`, `"nosso grupo"`
- `"[Nome] e [Nome] trabalhamos"`

**Mecanismo 2 — Sobreposição de nomes entre trabalhos:**
Para cada trabalho, extrai nomes próprios (maiúscula inicial, sem stopwords) e verifica quais correspondem a alunos no cadastro usando `difflib.get_close_matches`. Se aluno A menciona o nome do aluno B e aluno B menciona o nome do aluno A, candidatos ao mesmo grupo.

**Importante:** `difflib.get_close_matches` é biblioteca padrão (stdlib) — sem dependências externas novas. Consistente com ADR-004 e com a escolha de `SequenceMatcher` feita no ADR-006.

### Integração com plagiarism_detector

A modificação em `plagiarism_detector.py` é additive (parâmetro opcional com default `None`):

```python
def detectar_plagio_no_batch(
    trabalhos: list[Trabalho],
    threshold: float = 0.70,
    grupos_conhecidos: list[GrupoCandidato] | None = None,  # NOVO — AC-05
) -> list[ParPlagio]:
```

Lógica de supressão: antes de adicionar um `ParPlagio` à lista resultado, verificar se `aluno_a` e `aluno_b` pertencem ao mesmo `GrupoCandidato`. Se sim, suprimir. A supressão não altera notas nem flags — apenas omite o par da lista de suspeitos retornada.

### Convenção de RA

RAs são strings (definido pelo schema de Story 2.1 — `Aluno` model). O `group_detector` usa RA como identificador canônico dos membros, nunca o nome completo (nomes podem ter variações de grafia; RA é único e estável).

### Cenários mínimos obrigatórios

**TC-01 — `extrair_candidatos_grupo` encontra aluno mencionado por nome**
```
Arrange: texto com "eu e Bruno fizemos juntos"; aluno de RA "001" com nome "Bruno Silva"
Act: extrair_candidatos_grupo(texto, [aluno_bruno])
Assert: "001" in resultado
```

**TC-02 — `extrair_candidatos_grupo` tolera variação de grafia via fuzzy matching**
```
Arrange: texto com "Brunu me ajudou"; aluno de RA "001" com nome "Bruno"
Act: extrair_candidatos_grupo(texto, [aluno_bruno])
Assert: "001" in resultado  # "Brunu" → match fuzzy para "Bruno"
```

**TC-03 — `detectar_grupos_candidatos` retorna confiança "alta" para menção mútua**
```
Arrange:
  - Trabalho de Ana (RA "A01") menciona "Carlos"
  - Trabalho de Carlos (RA "C01") menciona "Ana"
Act: detectar_grupos_candidatos([trabalho_ana, trabalho_carlos], [aluno_ana, aluno_carlos])
Assert:
  - len(resultado) >= 1
  - resultado[0].confianca == "alta"
  - "A01" in resultado[0].membros and "C01" in resultado[0].membros
```

**TC-04 — `detectar_grupos_candidatos` retorna confiança "media" para menção unilateral**
```
Arrange:
  - Trabalho de Ana menciona "Carlos"
  - Trabalho de Carlos NÃO menciona "Ana"
Act: detectar_grupos_candidatos(...)
Assert: resultado[0].confianca == "media"
```

**TC-05 — Integração: `detectar_plagio_no_batch` suprime flag para par no mesmo grupo candidato**
```
Arrange:
  - 2 trabalhos com similaridade >= 0.70 (Ana e Carlos)
  - GrupoCandidato com membros = ["A01", "C01"]
Act: detectar_plagio_no_batch(trabalhos, grupos_conhecidos=[grupo_candidato])
Assert: pares retornados NÃO contém (Ana, Carlos)
```

**TC-06 — `extrair_candidatos_grupo` registra ambiguidade quando nome corresponde a múltiplos alunos**
```
Arrange:
  - texto com "João me ajudou"
  - 2 alunos com nome próximo: "João Silva" (RA "J01") e "João Costa" (RA "J02")
Act: extrair_candidatos_grupo(texto, [aluno_joao1, aluno_joao2])
Assert:
  - resultado (lista de RAs) está vazio ou não contém nenhum João
  - a evidência de ambiguidade é registrada: "Nome ambíguo: 2 alunos com nome similar encontrados"
```

**TC-07 — `detectar_grupos_candidatos` rebaixa confiança para "baixa" em grupos com mais de 4 membros**
```
Arrange:
  - 5 trabalhos onde cada aluno menciona os outros 4 (menção mútua plena)
Act: detectar_grupos_candidatos(trabalhos_5, alunos_5)
Assert:
  - resultado[0].confianca == "baixa"  # rebaixado pela regra de tamanho
  - len(resultado[0].membros) == 5
```

**TC-08 — `GrupoCandidato` popula `razao_confianca` com texto legível**
```
Arrange:
  - Trabalho de Ana menciona Carlos; trabalho de Carlos menciona Ana
Act: detectar_grupos_candidatos(...)
Assert:
  - resultado[0].razao_confianca != ""
  - "Ana" in resultado[0].razao_confianca or "Carlos" in resultado[0].razao_confianca
```

## Riscos

- **Risco 1:** Story 2.4 pode não estar concluída. Mitigação: se `plagiarism_detector.py` não existir, criar stub com assinatura mínima antes de implementar o grupo_detector. A modificação da Story 2.4 é additive (parâmetro com default None) — não quebra ACs existentes.
- **Risco 2:** Fuzzy matching de nomes curtos (ex.: "Ana") pode gerar falsos positivos em textos longos. Mitigação: usar `difflib.get_close_matches` com cutoff >= 0.75 e verificar que o match é com um token de tamanho compatível (>= 3 caracteres).
- **Risco 3:** Schema de `Aluno` (Story 2.1) pode diferir do esperado. Mitigação: verificar `src/` antes de implementar e ajustar imports. O campo de nome no `Aluno` model deve ter ao menos primeiro nome disponível.

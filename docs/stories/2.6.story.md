# Story 2.6 — Implementar `batch_state.py` — Estado intermediário sqlite3 efêmero

**Status:** Done
**Sprint:** 2 — Pipeline de batch de correção
**Complexidade:** S
**Prioridade:** Alta
**Dependências:** Pode ser implementada em paralelo com 2.2 ou antes dela — `batch_processor.py` (Story 2.2) importa este módulo. Sem dependência de outras stories Sprint 2.

---

## Contexto

O pipeline de batch de correção (Sprint 2) processa turmas inteiras de alunos chamando a API Anthropic em lotes de 10. Para tolerar falhas parciais sem reiniciar do zero, o `batch_processor.py` precisa de um estado intermediário que registre quais lotes foram processados, quais fichas foram salvas e quais lotes falharam.

Este módulo, `batch_state.py`, isola toda a lógica de banco de dados (sqlite3) do `batch_processor.py`. O isolamento tem dois benefícios diretos: (1) `batch_processor.py` pode ser testado com mocks de `batch_state` sem tocar sqlite3; (2) a lógica de persistência pode evoluir independentemente.

**Restrição de infraestrutura (Streamlit Cloud free tier):** o filesystem é efêmero e destruído a cada redeploy. O banco sqlite3 NÃO é persistência cross-session — é exclusivamente estado intermediário de sessão de batch. O path `/tmp/batch_state.db` é intencional. O padrão canônico obrigatório, definido no `SPRINT-2-INDEX.md`, é:

```python
# EFÊMERO: dados perdidos ao reiniciar sessão — comportamento esperado (ADR-004)
DB_PATH = Path("/tmp/batch_state.db")  # NUNCA usar caminho relativo ao projeto
```

**Contexto LGPD (ADR-004):** o schema do banco NÃO inclui nome do aluno — apenas `ra` (Registro Acadêmico) e o JSON da ficha de correção. Isso garante que os dados pessoais não sejam armazenados, mesmo que temporariamente em disco.

---

## Objetivo

Implementar `src/batch/batch_state.py` — módulo dedicado para gestão do estado intermediário de batch via sqlite3 efêmero em `/tmp/`. Isola toda lógica de banco de dados do `batch_processor.py`, tornando-o testável sem sqlite3 real.

## Acceptance Criteria

- [ ] **AC-01:** `DB_PATH = Path("/tmp/batch_state.db")` com comentário `# EFÊMERO:` exatamente conforme o padrão canônico do `SPRINT-2-INDEX.md` (comentário na linha anterior, texto exato: `# EFÊMERO: dados perdidos ao reiniciar sessão — comportamento esperado (ADR-004)`)
- [ ] **AC-02:** `inicializar_banco()` cria as 3 tabelas (`batches`, `lotes`, `fichas`) se não existirem — idempotente, seguro chamar múltiplas vezes sem erro
- [ ] **AC-03:** `salvar_lote(batch_id, lote_num, fichas)` persiste o JSON de cada ficha na tabela `fichas` com status `"processado"` — aceita lista de `FichaCorrecao`
- [ ] **AC-04:** `marcar_lote_falha(batch_id, lote_num, motivo)` registra o motivo na tabela `lotes` e marca status `"falha"` — NÃO lança exceção (falha silenciosa tolerada, pois o `batch_processor.py` já tratou o erro antes de chamar esta função)
- [ ] **AC-05:** `recuperar_fichas_batch(batch_id)` retorna todas as fichas de um batch como lista de `FichaCorrecao` (deserialização do JSON da coluna `ficha_json`)
- [ ] **AC-06:** `limpar_banco()` deleta TODOS os registros de TODAS as tabelas (`DELETE FROM batches`, `DELETE FROM lotes`, `DELETE FROM fichas`) — chamado ao iniciar novo batch para evitar acúmulo de dados em `/tmp/` durante a sessão
- [ ] **AC-07:** Toda abertura de conexão usa context manager `with sqlite3.connect(DB_PATH) as conn:` — sem conexões abertas residuais após cada operação
- [ ] **AC-08:** Schema da tabela `fichas` NÃO inclui coluna de nome do aluno — apenas `ra` e `ficha_json` (conformidade com ADR-004 — retenção zero de dados pessoais)
- [ ] **AC-09:** `pytest tests/unit/test_batch_state.py` passa usando `tmp_path` do pytest para sobrescrever `DB_PATH` via `monkeypatch` durante testes — sem tocar `/tmp/batch_state.db` real em disco
- [ ] **AC-10:** Todo bloco `except` neste módulo é tipado (ex.: `except sqlite3.Error as e:`, `except json.JSONDecodeError as e:`) — sem `bare except` nem `except Exception:` sem causa (conforme MNT-001 Sprint 1)

## Schema do Banco (tabelas obrigatórias)

### Tabela `batches`
Um registro por execução de batch:

| Coluna | Tipo | Notas |
|--------|------|-------|
| `id` | INTEGER PRIMARY KEY AUTOINCREMENT | |
| `sessao_id` | TEXT | Identificador da sessão Streamlit |
| `total_alunos` | INTEGER | Total de alunos no batch |
| `status` | TEXT | `"em_andamento"`, `"concluido"`, `"falha_parcial"` |
| `criado_em` | TEXT | ISO 8601, `datetime.utcnow().isoformat()` |

### Tabela `lotes`
Um registro por lote de 10 alunos:

| Coluna | Tipo | Notas |
|--------|------|-------|
| `id` | INTEGER PRIMARY KEY AUTOINCREMENT | |
| `batch_id` | INTEGER | FK para `batches.id` |
| `lote_num` | INTEGER | Número sequencial do lote (0-based) |
| `total` | INTEGER | Total de alunos no lote |
| `processados` | INTEGER | Quantos foram processados com sucesso |
| `status` | TEXT | `"processado"`, `"falha"` |
| `erro_msg` | TEXT | NULL ou mensagem de erro — preenchido por `marcar_lote_falha()` |

### Tabela `fichas`
Uma ficha por aluno processado:

| Coluna | Tipo | Notas |
|--------|------|-------|
| `id` | INTEGER PRIMARY KEY AUTOINCREMENT | |
| `batch_id` | INTEGER | FK para `batches.id` |
| `ra` | TEXT | Registro Acadêmico — NUNCA nome do aluno (ADR-004) |
| `status` | TEXT | `"processado"` |
| `ficha_json` | TEXT | JSON serializado da `FichaCorrecao` |
| `criado_em` | TEXT | ISO 8601 |
| `atualizado_em` | TEXT | ISO 8601 |

## Casos de Teste (mínimo 7)

### CT-01 — `inicializar_banco()` é idempotente
```
Chamar inicializar_banco() duas vezes seguidas não lança exceção
e as 3 tabelas existem ao final (usar PRAGMA table_list ou SELECT COUNT)
```

### CT-02 — `salvar_lote()` persiste fichas
```
Dado batch_id=1, lote_num=0, lista com 2 FichaCorrecao
Quando salvar_lote() é chamado
Então SELECT COUNT(*) FROM fichas WHERE batch_id=1 retorna 2
E cada linha tem ra correto e ficha_json deserializável
```

### CT-03 — `recuperar_fichas_batch()` deserializa corretamente
```
Dado fichas salvas via salvar_lote()
Quando recuperar_fichas_batch(batch_id) é chamado
Então retorna lista de FichaCorrecao com os mesmos valores salvos
```

### CT-04 — `marcar_lote_falha()` registra erro sem lançar exceção
```
Dado batch_id=1, lote_num=0, motivo="timeout na API"
Quando marcar_lote_falha() é chamado
Então não lança exceção
E SELECT status, erro_msg FROM lotes WHERE batch_id=1 AND lote_num=0
  retorna status="falha" e erro_msg="timeout na API"
```

### CT-05 — `limpar_banco()` remove todos os registros
```
Dado banco com dados em fichas, lotes e batches
Quando limpar_banco() é chamado
Então SELECT COUNT(*) de cada tabela retorna 0
```

### CT-06 — context manager sem leak de conexão
```
Após qualquer operação (salvar_lote, recuperar_fichas_batch, etc.)
Não há conexões abertas residuais — verificar via sqlite3.connect() abre normalmente
(DB não está bloqueado)
```

### CT-07 — schema sem coluna de nome de aluno
```
Verificar via PRAGMA table_info(fichas) que nenhuma coluna se chama
"nome", "nome_aluno", "name" ou variantes — apenas "ra" identifica o aluno
```

## Notas de Implementação

**Localização:** `src/batch/batch_state.py`

**Import do DB_PATH:**
```python
from pathlib import Path
import sqlite3
import json
from datetime import datetime

# EFÊMERO: dados perdidos ao reiniciar sessão — comportamento esperado (ADR-004)
DB_PATH = Path("/tmp/batch_state.db")  # NUNCA usar caminho relativo ao projeto
```

**Padrão de conexão obrigatório (AC-07):**
```python
def inicializar_banco() -> None:
    with sqlite3.connect(DB_PATH) as conn:
        conn.execute("CREATE TABLE IF NOT EXISTS ...")
        conn.commit()
```

**Serialização de FichaCorrecao:** usar `ficha.model_dump_json()` (Pydantic v2) ou `json.dumps(ficha.dict())` (Pydantic v1) — consultar como `FichaCorrecao` está definida no projeto antes de escolher.

**Monkeypatch nos testes:**
```python
def test_inicializar_banco(tmp_path, monkeypatch):
    db_path = tmp_path / "test_batch_state.db"
    monkeypatch.setattr("src.batch.batch_state.DB_PATH", db_path)
    inicializar_banco()
    # assertions...
```

**Tratamento de exceção tipado (AC-10):**
```python
# CORRETO
except sqlite3.Error as e:
    logger.warning("Falha ao marcar lote: %s", e)

# ERRADO — proibido por MNT-001
except Exception:
    pass
```

**Import de FichaCorrecao:** verificar o caminho correto no projeto. Provavelmente `from src.models.ficha_correcao import FichaCorrecao` — confirmar com `src/models/` antes de implementar.

## Riscos

- **Risco 1:** O modelo `FichaCorrecao` pode ainda não estar definido quando esta story for implementada (depende do andamento das stories de modelos Pydantic). Mitigação: se `FichaCorrecao` não existir, criar um stub com os campos mínimos (`ra: str`, `nota: float`, `feedback: str`) para desbloquear esta story — stub será substituído pela story de modelos.
- **Risco 2:** `src/batch/` pode não existir como pacote Python (sem `__init__.py`). Mitigação: verificar estrutura de `src/` antes de implementar e criar `__init__.py` se necessário.
- **Risco 3:** Permissões de escrita em `/tmp/` em ambiente de CI. Mitigação: os testes usam `tmp_path` via monkeypatch e NUNCA escrevem em `/tmp/batch_state.db` real — isso isola os testes do ambiente.

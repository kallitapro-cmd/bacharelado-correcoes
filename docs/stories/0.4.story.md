# Story 0.4 — ADR-003 — Definir política de temperature e determinismo da API Anthropic

**Status:** Done

## Contexto

O sistema usa a API Anthropic com dois modelos:
- **Haiku 4.5**: correção em batch (mais barato, maior volume)
- **Sonnet 4.6**: calibração e geração de feedback (mais capaz, menor volume)

A reprodutibilidade das notas é obrigatória por exigências de auditoria acadêmica: dado o mesmo trabalho e o mesmo critério de avaliação, o sistema deve produzir a mesma nota em execuções diferentes. Por outro lado, feedbacks com alguma variação ficam mais naturais para o aluno.

Essa tensão entre determinismo (notas) e variação controlada (feedbacks) precisa ser resolvida por política formal, não por experimentação ad-hoc no código.

## Objetivo

Criar o ADR-003 que define a política de temperature para cada fase do processamento, justificando as escolhas com base em requisitos de auditabilidade, custo e qualidade de resposta.

## Notas Técnicas

**Política base (definida na arquitetura v2.0, a formalizar no ADR):**
- `temperature=0` para geração de notas (determinismo total para auditoria)
- `temperature=0.3` para geração de texto de feedback (variação leve, mais natural)

**Fases esperadas do processamento:**

| Fase | Modelo | Propósito | temperature | max_tokens | Justificativa |
|------|--------|-----------|-------------|------------|---------------|
| Correção | Haiku 4.5 | Gerar nota + pontuação por critério | 0 | A definir | Auditoria exige reprodutibilidade |
| Calibração | Sonnet 4.6 | Revisar e ajustar notas geradas pelo Haiku | 0 | A definir | Consistência na calibração |
| Feedback | Sonnet 4.6 | Gerar texto de feedback para o aluno | 0.3 | A definir | Linguagem mais natural |
| Regeneração | Sonnet 4.6 | Regerar feedback quando solicitado pelo PA | 0.3 | A definir | Produz variação útil na regeneração |

**Implicação de auditoria:**
Temperatura 0 com o mesmo prompt e o mesmo modelo produz output idêntico (ou quase idêntico, dada a implementação da API). Isso permite ao PA demonstrar que a nota foi gerada pelo mesmo critério em caso de contestação.

**top_p:**
Avaliar se top_p deve ser definido explicitamente ou deixar no padrão do modelo. O ADR deve documentar a decisão (mesmo que seja "usar padrão").

**Relação com custo (referência cruzada):**
max_tokens impacta diretamente o custo. O ADR-003 deve referenciar o ADR-005 para que as estimativas de custo usem os valores de tokens definidos aqui.

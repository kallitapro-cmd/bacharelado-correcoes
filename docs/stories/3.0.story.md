# Story 3.0 — Tela de upload e configuração do batch

**Status:** Done
**Sprint:** 3 — Interface de validação do PA
**Complexidade:** M
**Prioridade:** Alta — gate de entrada do Sprint 3 (toda a interface depende desta tela)
**Dependências:** Sprint 2 completo (Stories 2.0–2.6 Done); `src/batch/batch_processor.py`; `src/batch/batch_state.py`; `packages/wrapper/clone_client.py`

---

## Contexto

O Sprint 2 entregou o pipeline completo de correção em batch (batch_processor, calibrator, plagiarism_detector, group_detector, batch_state). O Sprint 3 constrói a interface Streamlit para que o Professor Assistente (PA) interaja com esse pipeline sem precisar conhecer código.

Esta story implementa a **Tela 1** do fluxo desenhado pelo @oalanicolas (2026-05-22): o ponto de entrada onde o PA faz o upload da planilha de alunos, insere o enunciado do trabalho e configura os parâmetros do batch antes de iniciar o processamento.

O design prioriza **clareza radical**: poucos campos visíveis, sem jargão técnico, com validação imediata para evitar que o PA inicie um batch com dados inválidos.

---

## Objetivo

Implementar a tela de upload e configuração em `src/ui/upload_screen.py` como página Streamlit que valida a planilha, exibe preview dos dados e libera o botão "Iniciar correção" somente quando todas as pré-condições forem satisfeitas.

---

## Acceptance Criteria

- [ ] **AC-01:** A tela exibe uploader `st.file_uploader` aceitando apenas `.xlsx` e `.xls`; ao fazer upload de arquivo com extensão diferente, exibe `st.error` com mensagem clara e não prossegue.
- [ ] **AC-02:** Após upload válido, a tela exibe preview das primeiras 5 linhas do DataFrame com colunas mapeadas (RA, Nome, Turma); se colunas obrigatórias estiverem ausentes, exibe `st.error` com lista das colunas faltantes.
- [ ] **AC-03:** Campo `st.text_area` para o enunciado do trabalho (label: "Enunciado do trabalho", `max_chars=2000`); o botão "Iniciar correção" fica desabilitado enquanto o campo estiver vazio.
- [ ] **AC-04:** A tela exibe estimativa de custo calculada por `batch_processor.estimar_custo_brl()` atualizada em tempo real conforme o número de alunos detectado na planilha; exibe em formato `R$ X,XX`.
- [ ] **AC-05:** Se a estimativa de custo exceder `MAX_COST_BRL` (lido de `st.secrets` com fallback `15.0`), o botão "Iniciar correção" fica desabilitado e exibe aviso `st.warning` com o valor estimado vs. limite configurado.
- [ ] **AC-06:** O botão "Iniciar correção" só fica habilitado quando: (a) planilha válida carregada, (b) enunciado não-vazio, (c) estimativa dentro do orçamento. Ao clicar, grava em `st.session_state["batch_config"]` um dict com `{"alunos": list[dict], "enunciado": str, "metadata": dict}` e navega para a tela de progresso (Story 3.1).
- [ ] **AC-07:** Metadados capturados no `metadata` incluem: `turma` (input text), `professor` (input text), `data_prova` (date picker) — todos opcionais, mas incluídos no dict mesmo quando vazios (valores `""` ou `None`).
- [ ] **AC-08:** A tela NÃO inclui `nome` no dict `alunos` gravado em `st.session_state["batch_config"]` — o dict `alunos` contém apenas `{"ra": str, "turma": str, "resposta": str}` (ou campos equivalentes não-PII necessários para o squad); o campo `nome` é preservado SOMENTE em memória local para o preview da tabela e para ser reunido ao dict de exportação em `st.session_state["nomes_por_ra"]` como mapa `{ra: nome}` — jamais enviado à API (ADR-004).
- [ ] **AC-09:** Existe teste unitário `tests/unit/test_upload_screen.py` com pelo menos 3 testes: (a) planilha com colunas corretas → preview sem erro, (b) planilha sem coluna `ra` → erro de validação, (c) enunciado vazio → botão desabilitado (testado via lógica de validação pura, sem `st.testing`).

## Notas de Design (Alan Nicolas — 2026-05-22)

**Fluxo Tela 1 — Upload e Configuração:**
- Zone A (topo): upload do arquivo + preview imediato
- Zone B (meio): enunciado do trabalho + metadados opcionais (turma, professor, data)
- Zone C (rodapé): estimativa de custo + botão "Iniciar correção"
- Regra de ouro: botão habilitado = contrato visual de que tudo foi validado

**Princípio de clareza radical aplicado:**
- Sem abas, sem sidebars na tela de upload — fluxo linear de cima para baixo
- Feedback imediato após cada input (preview da planilha aparece sem precisar clicar)
- Estimativa de custo formatada em BRL (não USD, não tokens)

### Conformidade com ADRs

- **ADR-002 (Clone Contract)**: usa `BATCH_SIZE` importado de `packages.wrapper.clone_client` (fonte única de verdade).
- **ADR-004 (LGPD Retention Zero)**: `nome` excluído do dict `alunos` enviado ao squad; preservado apenas em `st.session_state["nomes_por_ra"]` para preview/exportação local.
- **ADR-005 (Budget Policy)**: gate de orçamento via `MAX_COST_BRL` com fallback seguro 15.0.

### Observações de Baixa Severidade (não bloqueantes)

1. **UI-001 — Formato monetário com ponto decimal**
   - `f"R$ {custo:.2f}"` produz `R$ 5.00` em vez de `R$ 5,00` (locale pt-BR).
   - AC-04 especifica literalmente "formato R$ X,XX". Funcionalmente correto, cosmeticamente desviante.
   - Ação sugerida: aplicar `.replace('.', ',')` ou `locale.currency()` em iteração futura.

2. **UI-002 — Captura ampla de exceções**
   - `except Exception as exc:  # noqa: BLE001` em `obter_max_cost()` (L94) e leitura de planilha (L173).
   - Mitigado: ambos têm fallback seguro e não expõem stack trace ao usuário (LGPD-safe).
   - Ação sugerida: restringir a `(ValueError, KeyError, OSError)` quando viável.

3. **UI-003 — Import math local**
   - `import math` dentro de `calcular_estimativa()` (L84) em vez do topo do módulo.
   - Trivial; mover para bloco de imports.

### Pontos Fortes

- Separação exemplar de responsabilidades: funções puras 100% testáveis sem `st.testing`.
- Validação defensiva em camadas (extensão de arquivo + colunas + tipos).
- Mensagens de erro claras e LGPD-safe (sem PII em logs/stack).
- 7x acima do mínimo de testes (21 vs 3 exigidos).
- Comentários explicativos referenciando ADR-004 nos pontos críticos.

# Story 0.3 — ADR-002 — Definir contrato de integração entre Wrapper Python e Squad Clone

**Status:** Done
**Sprint:** 0 — ADRs (pré-código)
**Complexidade:** M
**Prioridade:** Máxima
**Dependências:** nenhuma (pode rodar em paralelo com 0.1 e 0.2)

## Contexto

O sistema usa um squad existente em `squads/bacharelado-correcoes/` como fonte de inteligência para correção. A decisão arquitetural foi adotar um **Wrapper Python** em vez de um agente interativo, pois o Wrapper é mais previsível, mais barato e mais auditável para uso acadêmico.

O contrato de integração precisa ser formalizado porque:
1. O Wrapper Python precisa saber quais seções do `SKILL.md` compõem o system prompt.
2. A resposta da IA precisa ser um JSON estruturado para ser processado pelo Pydantic sem ambiguidade.
3. A política de retry e error handling precisa ser definida antes de qualquer código.

## Objetivo

Criar o ADR-002 que define o contrato completo entre o Wrapper Python e o squad `bacharelado-correcoes`, incluindo: mecanismo de carregamento do system prompt, schema JSON da resposta, política de retry e exemplos de chamada/resposta.

## Notas Técnicas

**Mecanismo do Wrapper (decisão já tomada na arquitetura v2.0):**
1. Wrapper lê o `SKILL.md` do squad em tempo de execução
2. Monta o system prompt a partir do conteúdo do arquivo (seções específicas a definir no ADR)
3. Chama a API Anthropic diretamente com esse system prompt
4. Mudanças no `SKILL.md` refletem automaticamente sem alterar código Python

**Schema esperado de response (a ser formalizado no ADR):**

```json
{
  "ra": "20260100418",
  "nota_a1": 8.5,
  "nota_a2": null,
  "matriz_pontuacao": {
    "criterio_1": 2,
    "criterio_2": 3
  },
  "feedback": "texto do feedback gerado pela IA",
  "flags": ["plagio", "ia_generativa"],
  "confianca": "alta"
}
```

**Campos a definir no ADR:**

| Campo | Tipo | Obrigatório | Valores válidos |
|-------|------|-------------|-----------------|
| `ra` | string | sim | RA normalizado (11 dígitos) |
| `nota_a1` | float ou null | sim | 0.0 a 10.0 ou null se não aplicável |
| `nota_a2` | float ou null | sim | 0.0 a 10.0 ou null se não aplicável |
| `matriz_pontuacao` | dict | não | chave: critério, valor: pontos |
| `feedback` | string | sim | texto livre, max a definir |
| `flags` | list[string] | sim | lista vazia se sem flags |
| `confianca` | string | sim | "alta", "media", "baixa" |

**Política de retry (sugestão a validar no ADR):**
- Tentativa 1: chamada normal
- Tentativa 2 (se resposta inválida): reenviar com instrução adicional de formato
- Tentativa 3: marcar como falha, adicionar à fila de revisão manual (Camada 3)
- Sem retry para erros de API (rate limit, timeout): propagar para o usuário

**Referência ao squad:**
- Localização: `squads/bacharelado-correcoes/`
- Arquivo principal: `squads/bacharelado-correcoes/SKILL.md`
- Seções relevantes para system prompt: a ser confirmado no ADR após leitura do SKILL.md atual

## Notas de Execução (Dev)

**Decisão de design — terminologia "SKILL.md":**
A Story 0.3 e o prompt original referem-se a um arquivo `SKILL.md` no squad. Verificação do squad em `squads/bacharelado-correcoes/` revelou que esse arquivo **não existe** com esse nome. O equivalente funcional é o conjunto `agents/corretor-academico.md` (entry-point, 793 linhas) + arquivos de suporte em `tasks/`, `data/`, `templates/`, `checklists/`. O ADR-002 formaliza explicitamente quais desses arquivos compõem o system prompt (8 incluídos, 4 excluídos com justificativa) e inclui uma "Nota terminológica" deixando essa equivalência clara para evitar confusão em stories futuras.

**Decisão de design — confiança no schema:**
O campo `confianca` (`alta`/`media`/`baixa`) não tem origem direta nos arquivos do squad — é uma adição do contrato Pydantic para sinalizar à Camada 3 (revisão manual) quais correções precisam de atenção humana prioritária. Documentado no ADR como "derivado das heurísticas (novo no contrato)".

**Decisão de design — exclusão de exportar-planilha do system prompt:**
A task `exportar-planilha.md` foi explicitamente excluída do system prompt porque concerne a etapa pós-correção (geração de planilha institucional), fora do escopo da IA de correção. Inclusão sobrecarrega tokens sem benefício comportamental.
